# 产品边界与用例

## 1. 产品定义

Notify Hub 是一个面向个人、家庭和小型自托管环境的通知与提醒平台。它同时提供：

- 企业微信应用消息收发；
- 外部系统事件接入；
- 单次、周期和持续催办提醒；
- 监控插件运行与管理；
- 通知去重、重试、投递记录和审计；
- 后续可扩展的图文、图片、语音和自然语言交互能力。

Notify Hub 不是通用自动化编排平台，也不是多租户商业 SaaS。第一阶段目标是：在一个可长期维护的自托管服务中，统一管理各种“发现事件并通知”的任务。

## 2. 角色

### 2.1 管理员

管理员通过 Web 后台：

- 配置企业微信应用；
- 管理企业微信成员映射；
- 创建和吊销 API Client；
- 安装、配置、启停和立即运行插件；
- 查看通知、事件、插件执行和错误日志；
- 管理提醒与持续催办任务；
- 执行连通性测试和数据维护。

第一阶段仅支持一个管理员账户。数据库设计保留未来多管理员能力，但不作为 MVP 验收项。

### 2.2 企业微信成员

企业微信成员可以：

- 接收指定给自己的通知；
- 后续通过文字或语音创建提醒；
- 查询、延后、完成或取消自己的提醒；
- 对持续催办任务进行确认；
- 使用应用菜单访问常用命令。

企业微信成员身份使用企业微信 `UserID`，不得使用昵称作为唯一标识。

### 2.3 API Client

API Client 是外部系统或脚本的调用身份，例如：

- GitHub Actions；
- Home Assistant；
- NAS 脚本；
- VPS 监控；
- 其他自建服务。

每个 Client 拥有独立 API Key、来源名称、权限和限流配置。不同 Client 不应共用同一个密钥。

### 2.4 监控插件

插件运行在 Notify Hub 的受控插件宿主中，负责连接特定数据源并产生标准化事件。插件不持有企业微信密钥，不直接发送消息。

## 3. 核心用例

### UC-01：外部事件通知

1. 外部系统调用 `POST /api/v1/events`；
2. API Key 鉴权通过；
3. 核心根据 `client_id + event_key` 执行幂等检查；
4. 事件先写入数据库；
5. 根据事件请求和路由策略创建投递任务；
6. Worker 发送企业微信消息；
7. 记录每个接收人的投递结果。

验收要求：调用方收到 `202 Accepted` 后，即使服务进程随后重启，事件仍可继续投递。

### UC-02：插件发现事件

1. 调度器触发插件运行；
2. 插件读取自己的配置、Secret 和状态；
3. 插件查询数据源；
4. 插件发现新事件后调用 `context.emit_event()`；
5. 核心执行与外部事件相同的去重和投递流程；
6. 插件保存采集游标；
7. 插件执行结果写入运行历史。

插件只有在核心成功接受事件后，才能推进“已处理游标”。

### UC-03：指定成员通知

调用方或插件指定一个或多个内部收件人 ID，核心解析为企业微信 `UserID`：

- 单人：`Vigoss`；
- 多人：核心转换为企业微信要求的分隔格式；
- 全部：明确请求 `@all`，且调用身份具备广播权限。

默认不允许插件随意向 `@all` 广播。

### UC-04：单次提醒

管理员或后续企业微信对话创建一个指定时间的提醒。提醒到期后产生内部事件并进入统一投递流程。

### UC-05：周期提醒

支持基于时区的周期规则。数据库存储规范化后的调度规则和下一次执行时间。服务重启后必须恢复。

### UC-06：持续催办

提醒可设置 `require_ack=true`：

- 首次发送后进入 `awaiting_ack`；
- 未确认时按间隔继续发送；
- 达到最大次数或截止时间后停止；
- 收件人确认后立即停止后续投递；
- 多收件人场景支持“任一确认”或“全部确认”。

### UC-07：文字创建提醒

企业微信成员发送自然语言文本。系统解析为提醒草稿，展示结构化结果并要求确认。确认前不得写入正式提醒表。

### UC-08：语音创建提醒

优先使用企业微信回调中的识别文本；没有识别结果时下载媒体并调用 ASR。其余流程与文字创建一致。

### UC-09：图文通知

通知可携带标题、正文、图片 URL 和跳转链接。渠道适配层负责转换为企业微信支持的消息类型。

### UC-10：语音通知

通知可携带已有语音文件，或由文本通过 TTS 生成音频。音频必须转换为企业微信支持的格式、上传临时素材后再发送。

## 4. 非目标

MVP 明确不做：

- 让用户从网页上传并立即执行任意 Python 包；
- 在主进程中以 `shell=True` 执行任意命令；
- 面向公网匿名开放通知接口；
- 多租户计费、套餐和组织隔离；
- 复杂工作流编排、可视化 DAG；
- 把 Notify Hub 做成 Home Assistant 或 n8n 的替代品；
- 让 LLM 直接决定数据库写入或执行系统命令；
- 依赖 MoviePilot 或 family-finance 才能运行。

## 5. 产品边界

### 核心平台负责

- 身份认证与权限；
- 事件接收和规范化；
- 事件幂等；
- 接收人解析；
- 通知格式化；
- 企业微信投递；
- 投递重试；
- 通知历史；
- 提醒生命周期；
- 插件注册、配置、调度和运行历史；
- Secret 管理；
- Web 管理界面。

### 插件负责

- 数据源协议；
- 采集频率建议；
- 数据源游标；
- 来源特有的判定逻辑；
- 来源特有的结构化数据提取；
- 产生事件，不负责发送消息。

### 外部脚本负责

没有必要开发为插件的外部系统，可以直接调用事件 API。它们负责自己的业务检测，Notify Hub 负责投递。

## 6. 成功标准

第一阶段完成时，应满足：

1. 新部署实例可以在 30 分钟内完成企业微信配置并发送测试消息；
2. Codex X Monitor 连续运行 7 天不重复通知同一推文；
3. Notify Hub 重启后不会丢失待投递消息和插件游标；
4. 企业微信临时故障后可自动重试；
5. 管理后台可以看到事件从“接收”到“投递成功/失败”的完整链路；
6. 任一插件异常不会导致 API 和其他插件停止工作；
7. 所有敏感字段不会出现在普通日志、前端响应或 Git 仓库中。
