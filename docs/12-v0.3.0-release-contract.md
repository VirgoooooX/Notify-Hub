# v0.3.0 发布契约

本文冻结 `v0.3.0` 的累计交付边界，并补足 Phase 7～9 在早期设计文档中尚未确定的实现细节。若与专题示例存在路径差异，以本文和 `docs/04-api-contracts.md` 的管理员 API 前缀为准。

## 1. 累计范围

`v0.3.0` 包含路线图 Phase 0～9：可靠事件投递、企业微信出站、插件与 Codex X Monitor、管理后台、文字/周期/持续提醒、企业微信回调，以及图片、语音输入、ASR、TTS 和语音投递。

图片入站 OCR、任意脚本执行、在线安装不可信插件、内置大模型权重、多实例和消息中间件不在本版本范围内。

## 2. 契约补充

- 首次和重复的 `POST /api/v1/events` 均返回 `202 Accepted`；重复请求返回原 `event_id` 和 `duplicate=true`，不创建新通知或投递。
- 管理员认证使用 `/api/v1/admin/auth/initialize|login|refresh|logout|me`，与 API Client Key 完全分离。
- Reminder 的 `awaiting_ack` 是由 `active + require_ack + 已生成至少一次投递` 推导的展示状态，不新增持久化状态。
- `any` 在任一合法接收人确认后完成整个 Reminder；`all` 和 `each` 均停止已确认接收人的催办并在所有接收人终止后完成，`each` 额外保留逐人结果用于业务汇总。
- Reminder 触发必须先产生 Event；Notification 的 `reminder_id` 仅作为查询优化和竞态二次检查的关联字段。
- `PluginRun` 同时作为持久化运行队列；立即运行只创建 `queued` 记录，Worker claim 后变为 `running`。
- 管理员 Reminder API 统一使用 `/api/v1/admin/reminders`。
- 持续催办使用 `template_card`；该值加入 Notification message type。
- 交互动作 Token 由随机 Action ID 与服务端 HMAC 在构卡时重建；Event/Notification 只持久化 Action ID，数据库不保存可直接使用的明文 Token。
- 企业微信运行时凭据只从环境变量或只读 Secret 文件注入；管理后台仅显示配置状态，避免数据库显示值与已启动 Adapter 分叉。

## 3. 媒体与语音

- 管理员媒体接口为 `/api/v1/admin/media`；通知通过 `media_asset_id` 引用受控媒体，不接受任意本地路径。
- 默认图片限制为 2 MiB、JPG/PNG；默认企业微信语音限制为 2 MiB、AMR、最长 60 秒。所有上限可通过配置进一步收紧，不能放宽到渠道不支持的范围。
- 外部媒体下载只允许 HTTP(S)，限制响应大小和重定向，并在 DNS 解析、实际连接地址及每次重定向后阻止 loopback、link-local、私网和云元数据地址。
- ASR/TTS 使用 Adapter。基础镜像不携带模型权重；可配置的本地命令 Adapter 以 argv 方式启动，不使用 shell，并有总超时、输出限制和临时目录隔离。
- 企业微信 `Recognition` 存在时优先使用；否则才下载语音并调用 ASR。识别结果只进入草稿流程，用户确认前不得创建 Reminder。
- TTS、转码、上传或 voice 发送失败时，只有在确认 voice 未成功后才创建一次文本降级投递，避免语音成功但响应丢失导致双发。
- 企业微信临时素材缓存必须保存过期时间；过期后重新上传。临时媒体默认不进入备份并按到期时间分批清理。

## 4. 发布门禁

发布前必须通过：空库 Alembic 升级；后端单元与集成测试；插件契约和固定 fixture；前端测试、类型检查与构建；Secret 泄露检查；事件幂等、Worker 租约恢复、提醒重启恢复；回调验签/解密/重放测试；恶意 MIME、超限媒体、SSRF；ASR 失败不建提醒；TTS 失败单次文本降级。

真实企业微信的文本、交互卡片、图片、语音和回调仍需在配置真实凭据的环境中完成人工验收。本机没有 Docker 时，容器构建与 Compose 启动由 CI 或部署主机验收。
