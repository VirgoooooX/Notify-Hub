# 架构决策记录

本文记录当前阶段已经确定的关键决策。后续若要改变，应新增决策条目并说明迁移方案，不应静默修改原有边界。

---

## ADR-001：采用模块化单体

**状态：已接受**

### 决策

Notify Hub 初期使用单仓库、单后端、单前端、单数据库、单 Docker 镜像。

### 原因

- 个人/家庭使用规模不需要微服务；
- 部署和备份简单；
- 插件、通知、提醒需要共享一致事务和管理界面；
- 通过模块接口仍可保持清晰边界；
- 后续可拆 API、Worker 和 Plugin Worker。

### 后果

第一阶段只能运行一个 SQLite 实例；插件异常隔离必须由 Plugin Runtime 完成。

---

## ADR-002：核心平台与监控插件同仓同部署

**状态：已接受**

### 决策

不再单独维护 `monitor-pack`。监控能力以插件形式位于 Notify Hub 仓库并由平台调度。

### 原因

- 一个管理后台；
- 一个部署单元；
- 统一配置、Secret、状态和日志；
- 更适合持续增加小型监控需求。

### 约束

同仓不代表直接耦合。插件必须通过 PluginContext 和 EventService 与核心交互。

---

## ADR-003：插件只发现事件，不直接发送通知

**状态：已接受**

### 决策

插件调用 `context.emit_event()`；企业微信发送、接收人解析、去重和重试由核心完成。

### 原因

- 避免每个插件重复实现渠道；
- 统一权限和审计；
- 渠道变化不影响插件；
- 防止 Secret 扩散；
- 保证可靠队列和投递记录。

---

## ADR-004：事件、通知、投递三层模型

**状态：已接受**

### 决策

- Event：发生了什么；
- Notification：准备发送什么；
- Delivery：发给谁以及当前状态。

### 原因

一个事件可能产生多个消息和多个接收人；每个接收人的发送结果应独立记录。

---

## ADR-005：事件先落库，异步投递

**状态：已接受**

### 决策

Event API 在 Event、Notification 和 Delivery 成功持久化后返回 `202 Accepted`。不在请求线程中依赖企业微信发送成功。

### 原因

- 企业微信临时故障不影响事件接收；
- 进程重启后可恢复；
- 可实现统一重试和历史；
- 调用方获得明确“平台已接管”的语义。

---

## ADR-006：初期使用 SQLite

**状态：已接受**

### 决策

MVP 使用 SQLite + WAL，限制单实例。

### 原因

- 自托管部署简单；
- 无额外数据库容器；
- 当前预期负载低；
- 备份恢复容易。

### 迁移条件

多实例、高写入竞争、高可用或复杂报表时迁移 PostgreSQL。

---

## ADR-007：独立企业微信“系统通知”应用

**状态：已接受**

### 决策

Notify Hub 使用独立企业微信应用，不复用 family-finance 或 MoviePilot 应用。

### 原因

- 来源清晰；
- 权限和 Secret 隔离；
- 任一业务系统升级不影响通用通知；
- 可独立配置可见范围和菜单。

---

## ADR-008：family-finance 和 MoviePilot 不做改造

**状态：已接受**

### 决策

两套系统继续负责各自业务和自身通知。Notify Hub 只承载新建的通用监控和提醒。

### 后续

若未来希望统一部分通知，可由它们调用 Notify Hub Event API，但不是 MVP 依赖。

---

## ADR-009：第一阶段只运行可信插件

**状态：已接受**

### 决策

插件来自仓库内置或管理员手工部署的审核代码，可在宿主进程中受控运行。暂不提供在线上传和任意安装。

### 原因

Python 进程内无法真正隔离恶意代码。先建立稳定插件契约，再在后续引入子进程/容器沙箱。

---

## ADR-010：插件调度和状态持久化

**状态：已接受**

### 决策

插件调度配置、运行历史和采集游标进入数据库。进程内调度对象不是事实来源。

### 原因

服务重启后必须恢复；后台需要查询；插件 emit 失败时不能丢失游标语义。

---

## ADR-011：幂等键由来源提供，数据库唯一约束兜底

**状态：已接受**

### 决策

唯一键为：

```text
(source_type, source_id, event_key)
```

Codex 插件使用帖子 ID，而不是内容哈希或当前时间。

### 原因

内容可能编辑、截断或跨来源格式不同；来源原生 ID 通常最稳定。

---

## ADR-012：提醒属于核心，不属于插件

**状态：已接受**

### 决策

单次提醒、周期提醒、持续催办、确认和延后由 ReminderService 负责。

### 原因

提醒需要统一的生命周期、收件人确认、时区和恢复机制，不应由每个插件重复实现。

---

## ADR-013：自然语言只产生草稿，确认后写正式提醒

**状态：已接受**

### 决策

规则解析器或 LLM 返回结构化 ReminderDraft；业务层校验并向用户展示，用户确认后才创建 Reminder。

### 原因

避免日期歧义、模型幻觉或错误接收人导致误提醒。

---

## ADR-014：先文本和图文，后语音

**状态：已接受**

### 决策

`v0.1.0` 实现文本和图文出站。语音输入、ASR、TTS 和 voice 消息放到后续版本。

### 原因

首版核心风险是可靠投递和插件系统，语音会引入媒体下载、转码、临时文件和模型依赖。

---

## ADR-015：推荐 FastAPI + Vue 3

**状态：已接受**

### 决策

- 后端：Python/FastAPI/Pydantic/SQLAlchemy/Alembic；
- 前端：Vue 3/TypeScript/Vite。

### 原因

- Python 适合监控插件和已有脚本迁移；
- FastAPI 便于异步 API、OpenAPI 和 Pydantic 契约；
- Vue 适合快速构建自托管后台；
- 与用户已有项目技术经验接近。

具体依赖版本在工程初始化 PR 中锁定，不在本决策文档写死。

---

## ADR-016：API Client 与管理员身份分离

**状态：已接受**

### 决策

外部事件使用独立 API Key，管理员后台使用独立认证。API Key 不能登录后台。

### 原因

权限、轮换、审计和限流语义不同，混用会扩大泄露影响。

---

## ADR-017：不提前引入 Redis 和消息中间件

**状态：已接受**

### 决策

MVP 使用数据库队列和持久化调度。只有出现明确瓶颈或多实例需求时再引入额外基础设施。

### 原因

减少部署复杂度和故障点；当前负载无需 Redis/RabbitMQ。

---

## ADR-018：首个插件为 Codex X Monitor

**状态：已接受**

### 决策

使用最初的 X/Twitter 监控需求验证插件、状态、事件、去重、调度、企业微信投递和后台管理完整闭环。

### 完成标准

同一帖子不重复、首次 baseline 不误发、平台失败不推进游标、重启后状态恢复。
