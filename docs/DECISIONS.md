# 架构决策记录

本文记录当前阶段已经确定的关键决策。后续若要改变，应新增决策条目并说明迁移方案，不应静默修改原有边界。

---

## ADR-001：采用模块化单体

**状态：已接受**

### 决策

Notify Hub 初期使用单仓库、单后端、单前端、单数据库、单 Docker 镜像。

### 原因

- 个人/家庭使用规模不需要微服务；
- 部署和备份简单；
- 插件、通知、提醒需要共享一致事务和管理界面；
- 通过模块接口仍可保持清晰边界；
- 后续可拆 API、Worker 和 Plugin Worker。

### 后果

第一阶段只能运行一个 SQLite 实例；插件异常隔离必须由 Plugin Runtime 完成。

---

## ADR-002：核心平台与监控插件同仓同部署

**状态：已接受**

### 决策

不再单独维护 `monitor-pack`。监控能力以插件形式位于 Notify Hub 仓库并由平台调度。

### 原因

- 一个管理后台；
- 一个部署单元；
- 统一配置、Secret、状态和日志；
- 更适合持续增加小型监控需求。

### 约束

同仓不代表直接耦合。插件必须通过 PluginContext 和 EventService 与核心交互。

---

## ADR-003：插件只发现事件，不直接发送通知

**状态：已接受**

### 决策

插件调用 `context.emit_event()`；企业微信发送、接收人解析、去重和重试由核心完成。

### 原因

- 避免每个插件重复实现渠道；
- 统一权限和审计；
- 渠道变化不影响插件；
- 防止 Secret 扩散；
- 保证可靠队列和投递记录。

---

## ADR-004：事件、通知、投递三层模型

**状态：已接受**

### 决策

- Event：发生了什么；
- Notification：准备发送什么；
- Delivery：发给谁以及当前状态。

### 原因

一个事件可能产生多个消息和多个接收人；每个接收人的发送结果应独立记录。

---

## ADR-005：事件先落库，异步投递

**状态：已接受**

### 决策

Event API 在 Event、Notification 和 Delivery 成功持久化后返回 `202 Accepted`。不在请求线程中依赖企业微信发送成功。

### 原因

- 企业微信临时故障不影响事件接收；
- 进程重启后可恢复；
- 可实现统一重试和历史；
- 调用方获得明确“平台已接管”的语义。

---

## ADR-006：初期使用 SQLite

**状态：已接受**

### 决策

MVP 使用 SQLite + WAL，限制单实例。

### 原因

- 自托管部署简单；
- 无额外数据库容器；
- 当前预期负载低；
- 备份恢复容易。

### 迁移条件

多实例、高写入竞争、高可用或复杂报表时迁移 PostgreSQL。

---

## ADR-007：独立企业微信“系统通知”应用

**状态：已接受**

### 决策

Notify Hub 使用独立企业微信应用，不复用 family-finance 或 MoviePilot 应用。

### 原因

- 来源清晰；
- 权限和 Secret 隔离；
- 任一业务系统升级不影响通用通知；
- 可独立配置可见范围和菜单。

---

## ADR-008：family-finance 和 MoviePilot 不做改造

**状态：已接受**

### 决策

两套系统继续负责各自业务和自身通知。Notify Hub 只承载新建的通用监控和提醒。

### 后续

若未来希望统一部分通知，可由它们调用 Notify Hub Event API，但不是 MVP 依赖。

---

## ADR-009：第一阶段只运行可信插件

**状态：已接受**

### 决策

插件来自仓库内置或管理员手工部署的审核代码，可在宿主进程中受控运行。暂不提供在线上传和任意安装。

### 原因

Python 进程内无法真正隔离恶意代码。先建立稳定插件契约，再在后续引入子进程/容器沙箱。

---

## ADR-010：插件调度和状态持久化

**状态：已接受**

### 决策

插件调度配置、运行历史和采集游标进入数据库。进程内调度对象不是事实来源。

### 原因

服务重启后必须恢复；后台需要查询；插件 emit 失败时不能丢失游标语义。

---

## ADR-011：幂等键由来源提供，数据库唯一约束兜底

**状态：已接受**

### 决策

唯一键为：

```text
(source_type, source_id, event_key)
```

Codex 插件使用帖子 ID，而不是内容哈希或当前时间。

### 原因

内容可能编辑、截断或跨来源格式不同；来源原生 ID 通常最稳定。

---

## ADR-012：提醒属于核心，不属于插件

**状态：已接受**

### 决策

单次提醒、周期提醒、持续催办、确认和延后由 ReminderService 负责。

### 原因

提醒需要统一的生命周期、收件人确认、时区和恢复机制，不应由每个插件重复实现。

---

## ADR-013：自然语言只产生草稿，确认后写正式提醒

**状态：已接受**

### 决策

规则解析器或 LLM 返回结构化 ReminderDraft；业务层校验并向用户展示，用户确认后才创建 Reminder。

### 原因

避免日期歧义、模型幻觉或错误接收人导致误提醒。

---

## ADR-014：先文本和图文，后语音

**状态：已接受**

### 决策

`v0.1.0` 实现文本和图文出站。语音输入、ASR、TTS 和 voice 消息放到后续版本。

### 原因

首版核心风险是可靠投递和插件系统，语音会引入媒体下载、转码、临时文件和模型依赖。

---

## ADR-015：推荐 FastAPI + Vue 3

**状态：已接受**

### 决策

- 后端：Python/FastAPI/Pydantic/SQLAlchemy/Alembic；
- 前端：Vue 3/TypeScript/Vite。

### 原因

- Python 适合监控插件和已有脚本迁移；
- FastAPI 便于异步 API、OpenAPI 和 Pydantic 契约；
- Vue 适合快速构建自托管后台；
- 与用户已有项目技术经验接近。

具体依赖版本在工程初始化 PR 中锁定，不在本决策文档写死。

---

## ADR-016：API Client 与管理员身份分离

**状态：已接受**

### 决策

外部事件使用独立 API Key，管理员后台使用独立认证。API Key 不能登录后台。

### 原因

权限、轮换、审计和限流语义不同，混用会扩大泄露影响。

---

## ADR-017：不提前引入 Redis 和消息中间件

**状态：已接受**

### 决策

MVP 使用数据库队列和持久化调度。只有出现明确瓶颈或多实例需求时再引入额外基础设施。

### 原因

减少部署复杂度和故障点；当前负载无需 Redis/RabbitMQ。

---

## ADR-018：首个插件为 Codex X Monitor

**状态：已接受**

### 决策

使用最初的 X/Twitter 监控需求验证插件、状态、事件、去重、调度、企业微信投递和后台管理完整闭环。

### 完成标准

同一帖子不重复、首次 baseline 不误发、平台失败不推进游标、重启后状态恢复。

---

## ADR-019：v0.3.0 使用可插拔本地媒体能力

**状态：已接受**

### 决策

ASR、TTS 和音频转码通过 Adapter 接入。基础镜像不内置模型权重；本地实现以无 shell 的受限 argv 命令运行，并通过挂载提供模型或可执行程序。

### 原因

- 保持文本事件核心可部署、可降级；
- 避免把未选定版本和许可的大模型固化进镜像；
- 允许按设备能力选择 SenseVoice 或其他本地实现；
- 统一超时、输出限制、临时文件和错误归一化。

---

## ADR-020：持续提醒确认状态与事件幂等语义

**状态：已接受**

### 决策

- `awaiting_ack` 为派生展示状态，不扩展 Reminder 持久化枚举；
- `any` 任一确认即完成，`all/each` 在全部接收人终止后完成且已确认成员不再接收；
- Event 首次与重复提交均返回 `202`，重复返回原 ID；
- Reminder 触发始终先产生 Event，再路由为 Notification/Delivery。

### 原因

保持已有状态机和“平台已接管”HTTP 语义稳定，同时消除专题文档与核心领域模型之间的歧义。

---

## ADR-021：废除 Secret 与普通配置的物理隔离机制

**状态：已接受**

### 决策

废除先前必须将敏感 Secret（如主密钥、JWT 密钥、企业微信 Token/AES key 等）通过 Docker Secret 物理挂载文件独立部署的规定。允许并在 Docker Compose 中推荐使用直接通过环境变量传递 Secret 的方式（以 Shell 或同级目录 `.env` 文件插值代入），从而废除 `/run/secrets` 下的密文小文件读取逻辑。

### 原因

- 简化单机和家庭环境自托管的部署与配置工作，不再需要额外创建多个零散的物理 Secret 文件。
- Pydantic Settings 和 `entrypoint.sh` 已能直接支持从纯环境变量中高效读取对应的敏感参数。
- 只要限制 `.env` 及宿主环境本身的读取权限，对该定位的低风险单机部署应用而言，安全性是完全足够且可接受的。

### 后果

- 部署目录中不再推荐创建 `secrets/` 文件夹。
- entrypoint.sh 移除了专门的 `_FILE` 转环境变量转换逻辑。
- 环境变量必须小心保管，不得提交明文到公共版本库。

## ADR-022：AI 作为核心 Gateway 能力

**状态：已接受**

### 决策

Provider、API Key、模型 Profile、缓存、预算、结构化校验和调用日志属于核心平台。Profile 是可复用运行策略，定义单一能力类型、模型路由、生成/输出策略和成本可靠性约束；具体业务 Prompt、标签和字段留在插件。插件只能调用 Manifest 明确授权且能力匹配的 `context.ai` Profile，并提供业务 instruction 与待分析数据；插件不得保存或指定 Provider URL、Key、任意请求头或模型。

第一版使用 OpenAI-compatible Chat Completions 作为最低公共协议，并保留后续 Responses 协议扩展点。防提示注入、禁用工具和结构化 Schema 校验是 Gateway 强制规则，不作为 Profile 开关。Provider 和 Profile 使用软删除保留调用审计；Provider 被未删除 Profile 引用时禁止删除，Profile 被启用插件引用时禁止删除。AI 返回结构化建议，Event 去重、是否发送和 checkpoint 仍由确定性代码负责。

### 原因

- 更换 Provider 或模型不需要逐个修改插件；
- 密钥、SSRF、预算、缓存和审计可集中治理；
- AI 故障不会绕过可靠事件与投递边界；
- 同一输入可跨插件运行和进程重启复用缓存，减少调用费用。
